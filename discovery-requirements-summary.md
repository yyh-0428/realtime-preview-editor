# realtime-preview-editor 需求摘要

> 项目名称：Werkstatt（匠人工坊）— 实时预览编辑器
> 形态：单 HTML 文件（~260KB），纯前端，本地运行
> 分析师：discovery-analyst（许明需）
> 日期：2026-06-13

---

## 一、五要素需求摘要

### 1. 场景（Surface）

| 场景类型 | 具体描述 | 使用频次 |
|---------|---------|---------|
| **快速原型验证** | 前端开发者用自然语言描述需求，AI 即时生成可运行的 HTML/JS 原型，右侧 iframe 实时预览效果 | 高频 |
| **AI 辅助编程教学** | 设计师/初学者通过对话式 AI 学习代码，观察"代码变化 → 视觉结果"的实时映射 | 中频 |
| **动效快速实验** | 利用内置 GSAP 全插件 + 设计系统融合，快速测试页面加载动画、滚动揭示、SVG 变形等效果 | 中频 |
| **数据可视化原型** | 调用内置 80+ Public API，快速搭建带真实数据的前端 demo（天气、金融、地图等） | 中频 |
| **设计系统探索** | 输入关键词（如"苹果风格登录页"），AI 自动从 16 套设计系统中推断 top-3 模板并智能融合 | 低频 |
| **离线/隐私敏感开发** | 本地单文件运行，API Key 自持有，无后端数据上传，适合涉密或网络受限环境 | 低频 |

**核心使用链路**：打开文件 → 输入自然语言指令 → AI 生成/修改代码 → 实时预览 → 微调迭代。

---

### 2. 受众（Audience）

| 用户画像 | 特征 | 核心诉求 | 使用习惯 |
|---------|------|---------|---------|
| **前端开发者（主力）** | 1-5 年经验，熟悉 HTML/CSS/JS，追求效率 | 快速出原型、减少样板代码、即时验证想法 | 偏好键盘操作，期望类似 CodePen/JSFiddle 的即时反馈 |
| **UI/UX 设计师** | 会基础代码，关注视觉效果和动效 | 无需写完整代码就能实现交互 demo | 偏好可视化结果，对代码细节不敏感 |
| **AI 辅助编程用户** | 使用 v0/Bolt/Cursor 等工具的早期采纳者 | 希望本地运行、数据隐私、更低成本 | 习惯自然语言描述需求，期望 AI 自主完成多步修改 |
| **独立开发者/创客** | 全栈但偏前端，项目周期短 | 从零快速生成可用页面、一键导出部署 | 重视工具链完整性和输出效率 |

**共同特征**：
- 对"实时反馈"有强依赖，延迟容忍度低（<500ms）
- 对 AI 生成代码的可控性有要求（需要 diff 确认、增量应用）
- 偏好轻量工具，排斥重 IDE 配置

### 3. 调性（Tone）

| 维度 | 当前定义 | 建议方向 |
|------|---------|---------|
| **品牌名** | Werkstatt（德语"工坊"） | 保留，强调工匠精神、手作感、精确性 |
| **视觉调性** | 暖色调暗色主题（Terracotta 赭石色点缀），低饱和度中性灰 | 持续强化"温暖的技术感"——区别于冷酷的深色科技风和泛滥的荧光绿 |
| **交互气质** | 细腻、有呼吸感（expo easing、微动效、状态过渡） | 保持，作为情感化设计的核心记忆点 |
| **文案语气** | 简洁、工具化、低侵入 | 保持，避免过度营销感 |
| **代码美学** | JetBrains Mono + 暖色语法高亮 | 保留，与 UI 的暖调形成统一 |

**一句话调性**："一把趁手的手工锉刀——不 flashy，但每一次交互都精确、温暖、可靠。"

### 4. 品牌上下文（Brand Context）

#### 竞品定位矩阵

| 竞品 | 核心优势 | 我们的差异化 | 我们的劣势 |
|------|---------|-------------|-----------|
| **v0.dev** | Vercel 生态闭环、组件级生成、一键部署 | 本地优先、设计系统融合更丰富、GSAP 全插件 | 无组件库沉淀、无一键部署 |
| **Bolt.new** | 多文件项目、StackBlitz 集成、npm 包安装 | 单文件零配置启动、更低门槛、数据隐私 | 无多文件支持、无 npm 生态 |
| **Lovable** | 自然语言精确度极高、产品级输出 | 设计系统多样性、动效能力更强 | AI 自主执行能力弱 |
| **Replit Agent** | 完整开发环境、数据库/后端支持 | 极轻量、无需注册、零延迟 | 无后端能力、无持久化项目 |
| **Cursor** | 多文件 Agent 能力、代码库级理解 | 可视化推理（深度思考卡片）、即时预览 | 无项目级索引、无多文件编辑 |

#### 核心差异化标签
1. **本地优先**（Local-First）：单文件、零依赖、数据不出本机
2. **设计系统融合引擎**：16 套内置风格，AI 自动推断 top-3 并融合
3. **GSAP 全插件生态**：预览环境预装全部 GSAP 插件，AI 可自动调用
4. **深度思考可视化**：SSE 流式展示 AI 结构化思考过程（竞品无此体验）
5. **80+ API 即开即用**：内置公共 API 知识库，AI 自动生成带真实数据的代码

### 5. 规模（Scale）

#### 当前状态
- **形态**：单页单文件工具（Single-File Utility）
- **架构**：纯前端 HTML + CSS + JS，无构建流程
- **数据持久化**：localStorage（代码、配置、片段）
- **AI 集成**：OpenAI-compatible API，单次 chat.completions 调用

#### 未来扩展方向（Roadmap）

| 阶段 | 目标 | 关键能力 |
|------|------|---------|
| **V1.0（当前）** | 单文件实时编辑 + AI 增量修改 | 代码编辑器、实时预览、AI Search/Replace、设计系统融合 |
| **V1.5（近期）** | 多文件虚拟项目 | 基于 localStorage 的虚拟文件系统、多 tab 切换、文件树 |
| **V2.0（中期）** | AI Agent 化 | ReAct 循环、轻量级 Tool Use（虚拟文件读写、预览刷新）、错误自修复 |
| **V2.5（中远期）** | 组件/模板生态 | 可视化组件库、用户模板市场、一键导出（ZIP/部署） |
| **V3.0（远期）** | 协作与扩展 | 多人实时协作、MCP 服务器集成、插件系统 |

## 二、用户痛点识别

### 高频痛点（每次使用都可能遇到）

| 痛点 | 严重程度 | 说明 |
|------|---------|------|
| **1. 无法从零创建项目** | 🔴 P0 | 当前只能基于已有代码修改，用户说"做一个天气仪表盘"时无法生成完整新页面 |
| **2. 单文件限制** | 🔴 P1 | 无法拆分为 HTML/CSS/JS 多文件，复杂项目代码膨胀、维护困难 |
| **3. 无 undo/redo 历史栈** | 🟡 P2 | AI 修改一旦应用无法回退，用户不敢尝试大胆指令 |
| **4. AI 单次调用局限** | 🔴 P1 | 复杂需求（如"修复所有响应式问题"）无法自动检查→验证→迭代 |
| **5. 代码无自动验证** | 🟡 P2 | AI 生成的代码可能有语法错误，需用户手动在预览中发现 |

### 中频痛点（每周可能遇到）

| 痛点 | 严重程度 | 说明 |
|------|---------|------|
| **6. 无组件/模板复用** | 🟡 P1 | 每次都要从头描述，无法复用之前生成的组件或社区模板 |
| **7. 设计系统选择门槛** | 🟡 P2 | 16 套设计系统对新手是认知负担，AI 推荐不够显性 |
| **8. 预览与编辑器状态不同步** | 🟡 P2 | 编辑器错误导致预览白屏时，无错误定位提示 |
| **9. 无一键分享/部署** | 🟡 P2 | 做好 demo 后只能通过下载 HTML 分享，无法生成在线链接 |
| **10. API Key 管理粗糙** | 🟡 P3 | 仅明文存储于 localStorage，无密钥掩码、多密钥管理 |

## 三、功能优先级建议

### 优先级矩阵

| 优先级 | 功能项 | 投入估算 | 预期收益 | 建议排期 |
|--------|--------|---------|---------|---------|
| **P0** | 从零生成项目（空白态 → AI 生成完整页面） | 1-2 天 | 极高：解锁全新用户场景 | 本周 |
| **P0** | 预览自动刷新（已完成） | — | — | ✅ 已完成 |
| **P1** | ReAct + 轻量 Tool Use（虚拟文件系统、多步推理） | 2-3 天 | 极高：AI 从"问答"升级为"助手" | 本周-下周 |
| **P1** | 多文件/多 tab 支持（虚拟文件树） | 2-3 天 | 高：解决单文件瓶颈 | 下周 |
| **P1** | 可视化组件库/模板画廊 | 3-4 天 | 高：降低重复工作、提升复用 | 第 2-3 周 |
| **P1** | undo/redo 历史栈（AI 修改快照） | 1-2 天 | 高：降低用户心理门槛 | 第 2 周 |
| **P2** | 前端代码验证层（HTML/JS 语法检查） | 1 天 | 中：减少预览错误 | 第 3 周 |
| **P2** | 一键导出/部署（ZIP、EdgeOne Pages 等） | 2 天 | 中：完善工作流闭环 | 第 3-4 周 |
| **P2** | 用户偏好学习（长期记忆） | 2 天 | 中：个性化体验 | 第 4 周+ |
| **P3** | 多 Agent 协作（Planner/Coder/Reviewer） | 1 周 | 高但投入大：接近 Cursor 体验 | 长期 |

### 推荐实施顺序

```
Week 1: 从零生成 + ReAct Tool Use 基础框架
Week 2: 多文件虚拟系统 + undo/redo 历史栈
Week 3: 组件模板画廊 + 代码验证层 + 一键导出
Week 4+: 多 Agent 协作 + 用户偏好学习 + 生态扩展
```

## 四、与主流竞品相比的致命短板

### 短板 1：AI 自主执行能力（最致命）

| 维度 | 我们 | Cursor/Bolt/Codex |
|------|------|-------------------|
| 推理模式 | 单次调用，提示工程模拟"深度思考" | ReAct 循环：思考 → 工具调用 → 观察 → 再思考 |
| 上下文范围 | 当前单一文件 | 整个项目、文件依赖图、符号索引 |
| 工具调用 | ❌ 无 | ✅ 文件读写、终端执行、浏览器预览、代码搜索 |
| 自修复能力 | ❌ 无 | ✅ 报错后自动分析并迭代修复 |

**影响**：用户无法下达复杂指令（如"给这个项目加用户认证"），AI 只能做简单局部修改。竞品已实现"说一句话，改十个文件"，我们仍是"说一句话，改一段代码"。

**弥补路径**：实施 ReAct + 虚拟文件系统工具调用（竞品分析文档方案 A，2-3 天投入）。

### 短板 2：项目级上下文缺失

- 当前 AI 每次只看到当前文件内容，看不到项目结构、依赖关系、历史修改意图
- 竞品（Cursor、Claude Code）会自动索引项目，建立代码知识图谱

**弥补路径**：多文件支持 + 导入语句自动分析 + 上下文优先级队列。

### 短板 3：无闭环部署能力

- v0 → Vercel 一键部署
- Bolt → StackBlitz 一键部署
- 我们 → 只能下载 HTML 文件

**影响**：用户做好 demo 后还需手动寻找部署方案，工作流断裂。

**弥补路径**：集成静态站点部署（如 EdgeOne Pages、GitHub Pages API）。

### 短板 4：无组件/模板生态

- 竞品有沉淀的 UI 组件库和社区模板市场
- 我们每次生成都是从零开始，成本高、一致性差

**弥补路径**：内置常用组件模板（导航、卡片、表单、仪表盘）+ 用户保存复用机制。

## 五、关键结论

1. **产品定位清晰但能力半径有限**：当前是一个优秀的"单文件 AI 辅助编辑器"，但距离"AI 编程助手"还有架构级差距。

2. **最大机会点在 ReAct + Tool Use**：纯前端限制无法执行真实系统命令，但基于 localStorage 的虚拟文件系统足以模拟多文件项目体验，这是投入产出比最高的升级。

3. **深度思考可视化是独特资产**：竞品分析显示，Cursor/Codex/Claude Code 均无结构化思考过程的可视化展示，这是我们的差异化体验，应持续强化。

4. **从零生成是用户准入门槛**：当前用户必须自带初始代码才能开始 AI 协作，这排除了大量"想做一个 XX 页面"的场景，必须尽快补齐。

5. **单文件既是限制也是护城河**：在隐私敏感、离线、快速验证场景下，"打开即用、无需注册、无网络依赖"是云原生竞品无法复制的优势，应在扩展功能时保留这一核心体验。
