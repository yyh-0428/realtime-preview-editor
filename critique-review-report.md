| 维度 | 得分 | 权重感受 |

## 维度一：哲学一致性（Philosophy）—— 3/5

### 评分理由

产品的核心理念被表述为"本地优先的轻量 AI 编辑器"。这一理念在三个层面出现了分裂：

**贯彻极好的层面——视觉与情感层**：
- Terracotta 赭石色点缀在深色画布上，确实营造出"温暖的技术感"，与冷酷的荧光绿科技风形成鲜明差异
- JetBrains Mono + DM Sans 的字体搭配统一且专业
- 微动效（expo easing、状态过渡、呼吸感）细腻不浮夸，符合"手工锉刀"的品牌隐喻
- 文案简洁工具化，无过度营销感

**出现矛盾的层面——架构与体验层**：
1. **"本地优先" vs 强网络依赖**：文件加载了 11 个外部 CDN 资源（CodeMirror、Prettier、Babel、Sass.js、highlight.js、marked、Google Fonts 等）。在无网络环境下，编辑器直接瘫痪。真正的本地优先应至少考虑核心依赖的内联或离线降级方案。
2. **"轻量" vs 260KB 单文件**：260KB 对于"轻量"一词已是临界值。打开文件时浏览器需要解析近 5000 行内联代码，低端设备上会有明显卡顿。
3. **"AI 编辑器" vs 单次调用局限**：产品定位暗示 AI 是"助手"，但当前架构只能做到"问答"。用户期望的 Agent 式协作与实际的单次 chat.completions 之间存在落差。

### 改进建议

- **短期**：为核心 CDN 资源添加 `integrity` 属性并准备本地 fallback，至少保证基础编辑能力离线可用
- **中期**：将设计系统数据、GSAP 知识库、Public API 知识库拆分为可选的按需加载模块（如 JSON 文件动态导入），而非全部硬编码在单文件中
- **长期**：重新定义"轻量"的度量标准——不应是文件大小，而是"用户从打开到产生第一个有价值输出的步数"

## 维度二：信息层次（Hierarchy）—— 3/5

### 评分理由

**清晰之处**：
- 经典的三栏布局（侧边栏 / 编辑器 / 预览）符合用户心智模型，CodePen/JSFiddle 用户零学习成本
- 面包屑式的 lang-tabs 在编辑器头部暗示多文件状态，虽然当前实现较为简陋
- 面板显隐切换的分段控件（segmented control）直觉性好
- 思考卡片（thinking card）的折叠/展开机制有效管理了信息密度

**混乱之处**：
1. **Toolbar 信息过载**：左侧已有品牌、主题切换、6 个主题色点、状态指示器；右侧有 3 个面板切换 + 6 个操作按钮（清空、复制、刷新、格式化、下载、导入）。共 18+ 个可交互元素挤在 44px 高的横条中，视觉权重扁平化——用户无法一眼判断"最重要的操作是什么"。
2. **AI 面板配置密度过高**：API 端点、API Key、模型选择、状态点、深度思考开关、设计系统下拉框——新用户首次展开 AI 面板时，看到的是一张"配置表单"而非"对话界面"。这与 ChatGPT/Claude 的"输入框即入口"极简哲学形成对比。
3. **设计系统选择器的认知负担**：16 个选项对前端专家都有压力，何况目标受众包含设计师和初学者。"AI 自动融合"虽然被设为默认，但下拉框中 16 个品牌的罗列仍构成选择 paralysis。
4. **空状态信息层级弱**：编辑器空状态时右侧预览 iframe 显示白屏或占位符，没有引导用户"第一步该做什么"。

### 改进建议

- **Toolbar 分级**：将"清空、复制、下载、导入"收入二级菜单（... 更多），Toolbar 只保留最高频的 3 个操作（格式化、刷新、AI 触发）
- **AI 面板渐进披露**：首次使用时只显示输入框和发送按钮，配置项折叠在"设置"抽屉中。参考 ChatGPT 的"先对话，后配置"策略
- **设计系统预筛选**：将 16 套缩减为 6 套高频 + "更多"扩展，或改用视觉缩略图网格替代文字下拉框
- **强化空状态引导**：在编辑器首次加载时，通过注释或占位符提示"按 Cmd+K 唤起 AI，或开始输入 HTML"

## 维度三：执行质量（Execution）—— 3/5

### 评分理由

**达到可用标准的精致功能**：
- **增量编辑引擎**：Search/Replace 格式解析、唯一匹配检测、多块排序替换、变更行高亮——这套机制在同类工具中属于上游水准
- **安全检查层**：代码长度比检测、HTML 标签缺失检测、JS 函数缺失检测——有效降低了 AI "幻觉删减"的风险，且支持自动回滚
- **SSE 流式可视化**：思考过程的实时流式渲染、状态机管理（`<thinking>` 标签解析）、节流刷新——技术实现扎实
- **设计系统自动推断**：基于关键词匹配的 top-3 推断 + 多模板融合提示词工程——虽然简单但有效
- **响应式与可访问性**：完整的移动端适配、reduced-motion 媒体查询、focus-visible  outline、ARIA 标签——体现了工程素养

**粗糙的边缘与关键缺失**：
1. **无 undo/redo 历史栈**：AI 修改一旦应用，用户只能依赖"安全检查失败时的自动回滚"来撤销。对于大胆尝试，用户缺乏心理安全感。这是 P2 级痛点但在代码中完全没有基础设施。
2. **无代码语法验证**：AI 生成的代码直接注入编辑器，没有 HTML/JS/CSS 的前置校验。预览 iframe 白屏时，用户无从得知是语法错误还是逻辑错误。
3. **AI 架构天花板明显**：所有精心设计的"深度思考"本质上仍是**单次提示工程**——通过 `<thinking>` 标签让模型自我独白，而非真正的多步 ReAct 循环。用户看到的"思考过程"是模型的自说自话，没有工具调用、没有观察反馈、没有迭代修正。
4. **多语言编译的脆弱性**：TS/SCSS 编译依赖运行时动态加载 Babel/Sass.js，失败时只有简单的错误 HTML，无行号映射，无法定位错误。
5. **预览错误静默失败**：iframe 内代码报错时，编辑器侧没有任何提示，用户需手动打开浏览器 DevTools。

### 改进建议

- **立即补齐 undo/redo**：为每次 AI 修改前后保存快照到 `localStorage` 数组，UI 上提供"撤销上一步 AI 修改"按钮——1-2 天可完成，收益极高
- **添加前端语法验证层**：HTML 用 `DOMParser`、JS 用 `new Function()`、CSS 用临时 `<style>` 标签试错——1 天可完成
- **iframe 错误捕获**：通过 `window.addEventListener('error')` 和 `message` 通道将 iframe 内错误回传至编辑器，显示行内提示
- **理性看待"深度思考"**：当前的 thinking 可视化是体验资产，但不应掩盖底层仍是单次调用的事实。应在 UI 中诚实标注"单次推理"，待 ReAct 实现后再升级为"多步思考"

## 维度四：特异性（Specificity）—— 3/5

### 评分理由

**真正独特的差异化资产**：
1. **设计系统融合引擎**：16 套内置设计系统 + AI 自动推断 top-3 + 智能融合提示——这是 v0/Bolt/Cursor 均不具备的能力。竞品通常只生成"默认现代风"或依赖用户明确指定框架。
2. **深度思考可视化**：SSE 流式展示 AI 的结构化思考过程（问题理解 → 需求分解 → 技术评估 → 方案分析...）。竞品分析确认 Cursor、Claude Code、Codex 均无此体验，是**真正的差异化记忆点**。
3. **本地优先 + 零注册**：在隐私敏感、离线、网络受限场景下，这是云原生竞品无法复制的护城河。

**陷入"什么都想做但什么都不精"陷阱的风险信号**：
1. **GSAP 全插件**：虽然产品在品牌中强调"GSAP 全插件内置"，但代码中并未发现 GSAP 的实际内联或 CDN 引入（仅在提示词中提到"用户项目已预装 GSAP"）。这是一个**虚假承诺**——预览 iframe 中并没有加载 GSAP，AI 生成的 GSAP 代码在预览中会因 `gsap is not defined` 而报错。
2. **80+ Public API**：同样只在提示词中作为知识库存在，没有实际的 API 调用封装、CORS 代理或数据缓存机制。AI 生成的 fetch 代码可能因 CORS 在 iframe 中失败，用户无法区分是代码错误还是网络限制。
3. **多语言支持（TS/SCSS/MD/JSON）**：功能列表漂亮，但 TS 编译依赖运行时加载 Babel（~250KB 额外下载），SCSS 依赖 Sass.js（~3MB WASM），在实际网络条件下体验不稳定。这些"支持"更像是 checkbox 功能，而非核心体验。
4. **与竞品的致命能力差距**：如需求摘要和竞品分析所述，AI 自主执行能力（ReAct 循环、工具调用、多文件编辑）与 Cursor/Bolt/Codex 存在代差。设计系统融合和 thinking 可视化是"锦上添花"，但 AI "能做什么"才是"雪中送炭"。

### 改进建议

- **诚实面对 GSAP**：要么在预览 iframe 的 srcdoc 中真正注入 GSAP CDN，要么从品牌宣传中移除该卖点。当前状态是功能虚假宣传。
- **收缩语言支持**：将 TS/SCSS 编译标为"实验性"，或将多语言支持降级为纯编辑器模式（无实时编译），避免不稳定体验损害核心口碑
- **放大真正独特的资产**：设计系统融合和 thinking 可视化应成为首页/空状态的核心卖点展示，而非 buried 在 AI 面板中
- **补上 ReAct 基础框架**：即使是最简化的虚拟文件系统工具调用（read_file / write_file / run_preview），也能将产品从"AI 问答工具"重新定位为"AI 助手"

## 维度五：克制（Restraint）—— 2/5

### 评分理由

这是五个维度中最弱的环节，也是制约产品长期健康发展的核心瓶颈。

**臃肿的量化证据**：
- **260KB 单文件**：4601 行代码中，CSS 约 1168 行，HTML 结构约 200 行，而 JavaScript 逻辑 + 数据约 3200+ 行
- **设计系统数据占用**：`DESIGN_SYSTEMS` 对象包含 16 套系统的完整颜色、字体、间距、圆角、规则、关键词——粗略估计占 60-80KB。`COMPONENT_MOTION_SPEC` 又追加多套组件/动效规范。这些数据是**静态的、只读的、极少被同时使用的**。
- **提示词工程过载**：`buildDeepThinkingPrompt` 根据复杂度分 3 档（low/medium/high），其中 high 档的 system prompt 长达 30+ 条思考步骤要求。`outputFormatRule` 又是大段格式约束。整个 AI 层充斥着"用提示词工程弥补模型能力不足"的焦虑感。
- **功能密度失控**：在一个单文件中，同时承载了：CodeMirror 编辑器、实时预览、8 语言支持、AI 聊天、SSE 流式、深度思考可视化、增量编辑引擎、安全检查、16 套设计系统、代码片段库、主题切换、响应式布局、面板拖拽、全屏预览、Prettier 格式化、文件导入导出、Modal/Toast/Confirm 组件系统... 这已经超过了一个"轻量工具"的合理边界。

**为了炫技而牺牲核心体验的具体表现**：
1. **6 个主题色点**：虽然视觉上精致，但"amber/slate/forest/ink/plum/ochre"与 16 套 AI 设计系统之间存在概念重叠。用户困惑："我选了 ochre 主题，又让 AI 用 Apple 设计系统，到底听谁的？"
2. **复杂度检测与分档思考**：`detectComplexity` 函数维护了两个庞大的关键词列表（high 190+ 词，simple 80+ 词），试图通过用户输入的关键词猜测复杂度。这是典型的过度工程——模型本身足以判断复杂度，或者用户可以通过一个显式开关选择。
3. **thinking 徽章的动画**：`badgeFadeIn` 动画、brainPulse 动画、dotPulse 动画、streamBlink 动画... 微动效数量之多，说明团队在"精致感"上投入了大量精力，但这些投入并未转化为用户核心任务效率的提升。

### 修正方向（明确）

由于本维度得分 < 3，给出以下明确修正方向：

**方向 A：数据与逻辑分离（立即执行，1-2 天）**
- 将 `DESIGN_SYSTEMS`、`COMPONENT_MOTION_SPEC`、`GSAP_KNOWLEDGE`、`PUBLIC_APIS_KNOWLEDGE` 提取为独立的 JSON 文件（如 `design-systems.json`），通过 `fetch()` 按需加载
- 目标：将主文件从 260KB 压缩至 120KB 以下，首屏加载时间减少 50%+
- 离线场景：在首次加载后缓存 JSON 到 localStorage，后续离线可用

**方向 B：功能分级与降级（1 周内）**
- 定义"核心体验环"（编辑器 + 预览 + AI 基础对话）和"扩展体验环"（多语言编译、深度思考、设计系统选择、代码片段库）
- 扩展体验环的功能在首次使用时以"解锁"或"启用"的方式呈现，而非默认全部展开
- 将 TS/SCSS 编译明确标为"实验性功能"，降低用户期望

**方向 C：提示词工程瘦身（1-2 天）**
- 将 3 档复杂度思考框架合并为 1 档通用框架（medium 档已足够），通过 `max_tokens` 和 `temperature` 控制输出长度，而非通过 system prompt 的篇幅
- 将 `outputFormatRule` 从 1500 字压缩至 300 字核心约束——模型对长约束的遵循率反而更低

**方向 D：主题系统收敛（立即执行）**
- 保留深浅色模式切换（必要）
- 将 6 个主题色点缩减为 2-3 个（如 Terracotta 默认、Slate 冷色备选、Forest 绿色备选），或将其移入设置菜单，避免 Toolbar 视觉噪音

## 跨维度关联诊断

五个维度的问题并非孤立，而是相互强化的：

```
克制不足（5） ──→ 单文件膨胀 ──→ "轻量"哲学受损（1）
     │
     ├─→ 功能堆砌 ──→ Toolbar/AI面板拥挤 ──→ 信息层次混乱（2）
     │
     ├─→ 提示词膨胀 ──→ AI单次调用token吃紧 ──→ 执行质量受限（3）
     │
     └─→ 虚假功能（GSAP/API） ──→ 特异性被稀释（4）
```

**核心矛盾**：团队将"单文件形态"视为无限容量的容器，而非约束条件。真正的单文件产品（如 Crafting Interpreters 的 Lox、早期 JSFiddle）懂得用外部资源加载、功能降级、数据懒加载来维持边界。Werkstatt 当前的状态是"把一款桌面应用的体量塞进了一个 HTML 文件"。

## 结论与优先级建议

### 一句话诊断

Werkstatt 是一个"交互精致、架构贪婪"的产品——视觉和微交互层面体现了工匠水准，但功能野心超出了单文件形态的合理承载力，导致核心体验被配置项、数据膨胀和提示词工程淹没。

### 最关键的三项修正（按优先级排序）

| 优先级 | 修正项 | 关联维度 | 投入 | 预期收益 |
|--------|--------|----------|------|----------|
| P0 | **数据外置**：将设计系统/知识库拆分为按需加载的 JSON | 克制、哲学 | 1-2 天 | 文件体积减半，首屏加载质变 |
| P0 | **补齐 undo + 前端验证**：AI 修改快照 + HTML/JS 语法预检 | 执行质量 | 1-2 天 | 用户心理安全感大幅提升 |
| P1 | **ReAct 基础框架**：虚拟文件系统工具调用 + 多步循环可视化 | 执行质量、特异性 | 2-3 天 | 从"问答工具"升级为"AI 助手"，补上与竞品最大差距 |

### 需要叫停的过度工程

1. **暂停新增设计系统**：16 套已足够，甚至过多。未来新增应走"用户自定义上传 JSON"路线，而非硬编码
2. **暂停新增语言支持**：在 TS/SCSS 编译稳定性达标前，不要扩展至 Vue/Svelte/Less 等
3. **简化深度思考框架**：3 档合并为 1 档，模型能力在持续进化，过度提示词工程是技术债
